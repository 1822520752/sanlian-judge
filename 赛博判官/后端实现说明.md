## 2. /api/profile —— B 站数据获取

### 实现要点
- 使用 `bilibili-api-python==17.4.2`
- 实际调用的 4 个方法:
  - `sync(u.get_user_info())` —— 昵称/头像/性别/签名/等级/VIP/认证
  - `sync(u.get_relation_info())` —— 粉丝数(`follower`)、关注数(`following`)
  - `sync(u.get_overview_stat())` —— 视频总数
  - `sync(u.get_videos())` —— 最近 10 个视频
- 字段映射(`v17` 实际结构 ≠ PRD 草图,做了兼容):
  - `regtime` ← `info.jointime` (v17 用 jointime,老库用 regtime)
  - `fans` ← `relation.follower`
  - `following` ← `relation.following`
  - `level` ← `info.level` (扁平 int;老库 `level_info.current_level` 兜底)
  - `vipType` ← `vip.type` (老库 `vip.vipType` 兜底)
  - `vipLabel` ← `vip.label.text`
  - `videos[].length` ← B 站已返回格式字符串 (`"03:34"`),兼容秒数
- `joinDays = floor((now - regtime) / 86400)`,regtime 缺失或非正时返回 0

### 异常分类
| 情况 | code | error |
|------|------|-------|
| UID 非法(非数字/超长) | -1 | `UID 只能输入数字` |
| B 站接口超时(>10s) | -1 | `B站接口开小差了` |
| B 站返回 -352 风控 | -1 | `B站触发风控，请稍后再试` |
| 用户不存在(-404) | -1 | `用户不存在` |
| 其它网络/未知异常 | -1 | `B站数据获取失败: <原因>` |

## 3. /api/analyze —— LLM 人格分析

### 实现要点
- `prompts/system.md` —— 整段抄自 PRD §6.1,固定不变
- `prompts/user.md` —— 整合 7 个模块的输入/输出/规则,使用 `{name}` `{fans}` 等占位符
- `api/_llm.py::call_deepseek()`:
  - 端点: `https://api.deepseek.com/v1/chat/completions`
  - model: `deepseek-chat`
  - `temperature: 0.9`,`max_tokens: 2000`
  - `response_format: {"type": "json_object"}`
  - 失败重试 3 次(指数退避 1s/2s/4s),5xx/429 也触发重试
  - 解析容错:
    1. 先 `json.loads(content)`
    2. 失败则去掉 ``` 围栏后再次 parse
    3. 再失败则在全文找首个平衡的 `{...}` 块 parse
  - 缺失 API Key → `ValueError("DEEPSEEK_API_KEY 未配置")`

### Prompt 渲染
`analyze.py::_render_user_prompt()` 把 profile 字段填入 `prompts/user.md` 的占位符
(uid / name / face / sex / sign / level / fans / following / vipType / vipLabel /
 official_json / regtime / joinDays / videos_json)。

### 响应合并与兜底
`analyze.py::DEFAULTS` 给 7 个模块都定义了完整的默认结构。
`_deep_merge_defaults()` 用 LLM 返回的字段覆盖默认,缺失的字段自动补上默认值
(`personaType.type` 默认为 `"B站普通用户"`,`craziness.score` 默认为 50 等)。

### 排行榜写入
LLM 成功返回后,会调用 `api/_rank_store.py::write_rank()`,
写入 `{uid, name, score=craziness.score, level, avatar, timestamp}`。
写入失败不阻塞主响应。

### 错误码
| 情况 | code | error |
|------|------|-------|
| 缺 uid | -1 | `缺少 uid 参数` |
| uid 非数字 | -1 | `UID 只能输入数字` |
| 缺 API Key | -1 | `DEEPSEEK_API_KEY 未配置` |
| LLM 超时 | -1 | `AI判官累了` |
| LLM 返回非 JSON | -1 | `AI 分析失败: ...` |
| 成功 | 0 | null |

## 4. /api/rank —— 排行榜

### 实现要点
- `api/_rank_store.py` 提供 `read_rank()` / `write_rank()`,**优先写 Vercel KV**:
  - 检测 `KV_REST_API_URL` + `KV_REST_API_TOKEN` 环境变量
  - KV 可用:走 `/get/<key>` `/set/<key>` REST 接口
  - 不可用:读写 `data/rank.json` 作为兜底
- 写逻辑:
  - 按 `uid` 去重
  - 按 `score` 降序排序
  - 保留前 50 条(防无限增长)
- 读逻辑:

## 5. Vercel Serverless 适配

每个 `api/*.py` 都定义了模块级 `handler(request)`,支持两种调用形式:

1. **线上 (Vercel 注入)** —— `request.query` / `request.body` 属性访问
2. **本地 (CLI 调试)** —— `handler({"query": {"uid": "..."}})` dict 形式

为了同时兼容两种入口,实现里都用了 `_parse_query()` / `_parse_body()` 工具函数,
依次尝试 `request.query` / `request.queryStringParameters` / `request.params` / `request.body` 等字段。

## 7. 已知遗留问题

1. **`regtime` / `joinDays` 对部分老用户可能为 0**
   B 站 v17 的 `get_user_info()` 对部分老账号(如老番茄)返回 `jointime: 0`,
   这属于 B 站侧数据问题,前端可显示为"元老级 UP 主"等兜底文案。
2. **`videos` 在 B 站风控时可能为空**
   高频调用或被 412 风控时 `get_videos()` 会抛 `NetworkException`。
   代码已 try/except 兜底返回空数组,不影响其它字段。
3. **Vercel Serverless 中 `data/rank.json` 写入可能不持久化**
   Vercel 的 `/tmp` 之外的文件系统是只读的,推荐配置 `KV_REST_API_*` 环境变量
   让排行榜走 KV;否则在 Vercel 上文件写入会被忽略,排行榜读到的将是预置的 10 条兜底。
4. **bilibili-api 第三方依赖**
   若部署时冷启动较慢,可考虑预装或迁移到自实现的 `requests` 调用(直接打 B 站公开 API),
   但 MVP 阶段先用现成库。
